### 들어가기 전에

6개월 전에 Claude Code(CC) 일주일 써보고 후기 남겼던 놈입니다. 이제 하드코어하게 쓴 지 6개월이 됐고, 그동안 쌓인 팁과 '말 대잔치'를 좀 풀어볼까 해요. 내용이 좀 길고 빡셀 수 있으니, 커피 한 잔 타 오시든 화장실에 앉으시든 마음의 준비 단단히 하세요.

미리 말씀드리자면, 이건 제가 지금 가장 잘 쓰고 있는 설정일 뿐이지 '절대 진리'는 아닙니다. 여러분의 AI 코딩 워크플로우에 영감을 주길 바랄 뿐이에요. 저도 그냥 한 명의 개발자고, 이건 그냥 제 개인적인 의견일 뿐이니까요.

참고로 저는 **20x Max 플랜** 사용자입니다. 그냥 대충 느낌대로 코딩하는 '바이브 코딩' 팁을 찾으신다면 다른 데 알아보세요. CC를 제대로 뽑아먹으려면 기획, 리뷰, 반복, 대안 탐색까지 AI와 **함께** 굴러야 합니다.

### 요약

혼자서 **30만 줄(LOC)을 재작성**하며 한계를 몰아붙인 끝에 구축한 시스템입니다.

* 필요할 때 알아서 켜지는 **자동 활성화 스킬(Skills)**
* Claude가 헛소리 못 하게 잡아주는 **개발 문서(Dev Docs) 워크플로우**
* 에러 하나 용납하지 않는 **PM2 + 후크(Hooks)**
* 리뷰, 테스트, 기획을 담당하는 **전문 에이전트 군단**

### 배경

7년 차 웹 엔지니어이고, AI라는 파도를 두 팔 벌려 환영하고 있습니다. 일자리 뺏길 걱정보다는, 이걸로 제 역량을 얼마나 뻥튀기할 수 있을지 고민하죠. 덕분에 예전 같으면 꿈도 못 꿨을 프로젝트들을 쳐내면서 회사에서 'AI 구루'로 자리 잡았습니다.

자신감이 넘쳤던 저는 직장에서 쓰는 내부 도구의 대대적인 리뉴얼을 제안했습니다. 7년 전 제가 인턴 때 만든 프로젝트를 누군가 4년 동안 누더기처럼 이어온 끔찍한 놈이었죠. 10만 줄짜리 프로젝트를 혼자서 2~3개월 안에 완전히 갈아엎겠다고 장담했습니다.

그리고 6개월이 지났습니다... 네, 그런 약속은 함부로 하는 게 아니었습니다. Claude와 제 정신줄의 한계를 시험했죠. React 16 JS를 React 19 TS로, 모든 라이브러리를 최신 버전으로 올리며 베스트 프랙티스를 떡칠했습니다. 결과물은 30~40만 줄로 불어났고 **제 기대 수명은 한 5년쯤 줄어든 것 같네요.** 하지만 결과는 끝내줍니다. 기술 부채 제로, 테스트 커버리지 완벽, 개발자 경험까지 다 잡았습니다.

### 품질과 일관성: 삽질은 그만

가끔 "요즘 Claude 품질 떨어진 거 아님?" 하는 소리가 들리는데, 제 경험상 워크플로우만 잘 다듬으면 품질은 오히려 올라갑니다.

**때로는 직접 나서세요.**
AI는 마법이 아닙니다. 30분 동안 Claude가 삽질하는 걸 보고 있느니, 내가 2분 만에 고칠 수 있다면 그냥 직접 고치세요. 자존심 세우지 마시고요. 자전거 가르칠 때 가끔 핸들 잡아주는 거랑 똑같습니다. "AI가 다 해줘야지"라는 생각에 갇혀서 시간 낭비하지 마세요.

프롬프트가 구리면 결과도 구립니다. `Esc` 두 번 눌러서 이전 프롬프트 불러오고 계속 다시 지시하세요. **"Claude가 나한테 뭘 해줄까 묻지 말고, 내가 Claude한테 어떤 컨텍스트를 줄 수 있는지 고민하세요."**

### 나의 시스템

#### 1. 스킬 자동 활성화 (이게 진짜 게임 체인저!)

Anthropic이 스킬 기능을 내놨을 때 "오 대박!" 했죠. 근데 문제가 뭐냐면, **Claude가 이 스킬들을 안 써요.** 가이드라인을 수천 줄 써놨는데, 설명서 안 읽고 가구 조립하는 사람처럼 제멋대로 코딩하는 거죠.

그래서 **후크(Hooks)**를 썼습니다. 안 읽으면 읽게 **만들면** 되죠.

* **UserPromptSubmit 후크:** Claude가 제 메시지를 읽기 전, 키워드를 분석해서 관련 스킬을 컨텍스트에 박아버립니다. (예: "🎯 스킬 활성화 체크 - 프로젝트 카탈로그 스킬 사용")
* **Stop Event 후크:** 응답이 끝난 뒤 위험한 패턴(에러 처리 누락 등)을 분석해서 "이거 확인했어?"라고 잔소리를 던집니다.

#### 2. 개발 문서(Dev Docs) 시스템: 기억력 금붕어 치료법

Claude는 자신감은 하늘을 찌르는데 기억력은 금붕어 같은 주니어 개발자입니다. 30분만 지나도 우리가 뭘 하려 했는지 까먹죠. 그래서 큰 작업을 할 땐 무조건 3개 문서를 만듭니다.

* **계획서(Plan):** 설계도
* **맥락노트(Context):** 왜 이렇게 결정했는지
* **체크리스트(Tasks):** 뭘 했고 뭘 할 건지

이걸 파일로 저장해두면 대화가 아무리 길어져도, 심지어 대화창을 새로 열어도 이 문서들만 읽히면 바로 작업 복구가 됩니다. **기획은 왕입니다.** 기획 모드에서 계획부터 제대로 세우고 들어가세요.

#### 3. PM2와 백엔드 디버깅

백엔드 마이크로서비스 7개를 동시에 돌리는데 로그 보는 게 고역이었죠. 이제 PM2를 써서 Claude가 직접 로그를 실시간으로 읽게 합니다. "메일 서비스 왜 터짐?" 물어보면 Claude가 알아서 `pm2 logs email` 치고 분석해서 고쳐놓습니다. 전 그냥 구경만 하면 돼요.

#### 4. "쓰레기는 남기지 않는다" 후크 시스템

Claude가 코딩해놓고 빌드도 안 돌려보고 퇴근하는 꼴은 못 봅니다.

* **빌드 체크 후크:** 응답 끝날 때마다 자동으로 빌드 돌려서 에러 잡습니다. 에러 남겨두고 "다 됐어요!" 하는 꼴 이제 안 봐도 됩니다.
* **Prettier 후크:** 코드 스타일 제멋대로인 거 못 참죠. 자동으로 포맷 맞춰줍니다. (단, 대형 파일에선 토큰을 많이 먹으니 주의하세요.)

### 결론

6개월간 굴러본 결과는 이겁니다.

1. **무조건 계획부터 짜라.**
2. **스킬 + 후크:** 자동 활성화만이 살길이다.
3. **개발 문서 시스템:** Claude의 뇌 기능을 외부 문서로 확장해라.
4. **리뷰:** Claude가 짠 코드는 다른 에이전트(AI) 시켜서 한 번 더 털어라.

AI는 그냥 쓰면 50점짜리지만, **시스템을 깔아주면 95점짜리 에이스가 됩니다.** 도구 탓하기 전에, 그 도구가 제대로 굴러갈 환경부터 만들어주세요.
